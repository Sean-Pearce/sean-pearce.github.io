命令行代理小工具——gkd

2019-07-21
技术
约 1320 字 预计阅读 3 分钟

哈罗大家好，转眼间已经到了七月份的尾巴，距离上次更新博客已经差不多两个月了，惭愧惭愧……

从 GitHub clone 过代码的童鞋肯定都知道，墙内拉代码的速度实在是太慢了，小一点的项目还好，如果代码仓库大小超过了 10MB，那可得等好一会儿了。 有时候运气不佳甚至有可能断线，然后就只能“少侠请重新来过”了。

但是作为一只专业的程序猿，怎么能被一面矮墙（并不）挡住去路呢。我们先在开发环境中配置代理（按下不表），然后修改一下 git 的配置：

1
2
3
4

# http proxy
git config --global http.https://github.com.proxy http://ip:port
# socks5 proxy
git config --global http.https://github.com.proxy socks5://ip:port

解释一下，上面两条命令是用来修改 git 的代理设置的，http. 到 .proxy 中间的字段可以指定要走代理的域名，如果留空的话就相当于全局代理。 最后面的 http://ip:port 就是代理地址了，两种代理协议的配置只有这里有区别。

矮哟不错哟，2.56MB/s 的平均速度基本上什么库都是秒下了。

既然问题已经解决了，那这篇文章是不是就到此为止了呢？非也非也~ git 的问题的确解决了，但 Linux 系统中还有很多其他的命令行工具需要代理，比如 wget，curl 这些， 怎么才能让它们也走代理呢？比较传统的办法是直接修改当前 shell 的环境变量：

1

export http_proxy=http://ip:port

但是这样做有一个问题：在你手动把环境变量改回来之前，后续执行的所有命令都会遵循这个设置（走代理），失去了灵活性。那么，有没有一种方法可以非常方便地指定一条命令走不走代理呢？

当然有啦，不然我也不会写（水）这篇文章了哈哈哈。接下来先讲怎么用，再讲为什么。

首先随便找个地方新建个文件，就叫 gkd 吧：

1

touch gkd

然后把下面的内容复制到这个文件中（记得修改代理地址）：

1
2
3

#!/usr/bin/env bash
export http_proxy=http://ip:port
exec ${@:1}

再赋予执行权限：

1

chmod +x gkd

最后，把这个脚本扔到任意一个 PATH 目录下，这里我选了 /usr/bin 目录

1

move gkd /usr/bin

好了，现在配置已经完成，只要在一条命令前加上 gkd，这条命令的网络流量就会走代理，而且对当前 shell 的环境变量没有任何影响。 我们来测试一下：

接下来解释一下这个 gkd 脚本的工作原理。

1
2
3
4
5
6
7
8

# 指定 shell 解释器为 bash。
#!/usr/bin/env bash

# 设置 http_proxy 环境变量
export http_proxy=http://ip:port

# 这里是关键
exec ${@:1}

exec 是 bash 中的一个内部命令，用于执行 ${@:1} 所表示的命令。${@:1} 表示传递给这个脚本的所有参数，当执行 gkd curl ipinfo.io 时，curl 和 ipinfo.io 是传递给这个脚本的两个参数，因此 exec ${@:1} 会被解释为 exec curl ipinfo.io，而且 exec 命令可以继承父进程的环境变量，从而实现代理。而执行脚本的进程本身是我们敲命令的 shell 的子进程，修改它的上下文不会影响父进程，因此代理的影响只局限于 gkd curl ipinfo.io 一条命令。
